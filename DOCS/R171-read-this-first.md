# R171 읽기 우선순위 정리

이 문서는 `DOCS/R171(kor).md`를 읽을 때, **먼저 읽어야 할 부분**과 **프로젝트와 직접 연결되는 부분**을 빠르게 찾기 위한 안내서다.

표기 규칙:

- `(**프로젝트 직접 연관**)` = `DCAS-PolicyEngine`의 Step B / Step C / HMI / RMF 해석에 바로 영향을 주는 조항
- `그 외` = 전체 맥락 이해를 위해 읽으면 좋은 조항

---

## 1) 제일 먼저 읽어야 할 핵심

- **2.5 시스템 경계(System Boundaries)** `(**프로젝트 직접 연관**)`
  - DCAS가 어디까지 지원 가능한지 정의하는 핵심 기준이다.

- **2.6 운전자 이탈(Driver disengagement)** `(**프로젝트 직접 연관**)`
  - Step B 상태 전이의 개념적 기준이다.

- **2.18 ~ 2.18.2.2 작동 모드(Off / On / Active / Stand-by / Passive / Inactive)** `(**프로젝트 직접 연관**)`
  - 상태머신과 HMI가 어떤 모드에서 무엇을 해야 하는지 이해하는 데 필요하다.

- **5.1 일반 요구 사항** `(**프로젝트 직접 연관**)`
  - 운전자 지속 참여, 모드 인지, 오버라이드 가능성, AEBS 연동의 기본 방향을 잡아준다.

- **5.5.3 활성화 / 비활성화 / 운전자 오버라이드** `(**프로젝트 직접 연관**)`
  - 언제 시스템이 켜지고 꺼져야 하는지, 운전자가 언제든 개입할 수 있어야 하는지의 기준이다.

- **5.5.4 운전자 이탈 감지 및 경고** `(**프로젝트 직접 연관**)`
  - WARNING / UNRESPONSIVE / ABSENT 설계의 직접 근거다.

---

## 2) Step B를 이해하려면 꼭 읽어야 하는 부분

- **5.5.4.2.5.1 ~ 5.5.4.2.5.3 시각적 이탈 평가 및 재참여** `(**프로젝트 직접 연관**)`
  - 재참여 판단 최소 지속시간(200ms)과 반복적인 짧은 이탈 대응 전략이 포함된다.

- **5.5.4.2.6.1 ~ 5.5.4.2.6.4 경고 에스컬레이션 시퀀스** `(**프로젝트 직접 연관**)`
  - HOR / EOR / DCA / driver unavailability response의 시간 구조를 이해해야 한다.

- **5.5.4.2.6.2.1, 5.5.4.2.6.2.2** `(**프로젝트 직접 연관**)`
  - 속도 10 km/h 초과 시 EOR 5초, 이어서 3초 후 에스컬레이션이라는 핵심 앵커가 있다.

- **5.5.4.2.6.3.1** `(**프로젝트 직접 연관**)`
  - EOR 에스컬레이션 후 5초 내 DCA가 요구된다.

- **5.5.4.2.6.4.1** `(**프로젝트 직접 연관**)`
  - 첫 에스컬레이션 후 10초 이내 driver unavailability response가 요구된다.

- **5.5.4.2.7 추가 전략 / 5.5.4.2.8 반복 또는 장기 이탈** `(**프로젝트 직접 연관**)`
  - 장시간 미응답, 반복적인 짧은 이탈, 시스템 재활성화 제한 같은 운영 전략을 이해하는 데 필요하다.

---

## 3) Step C / 제어 정책과 직접 이어지는 부분

- **5.3.4 제어 전략과 운전자 반응 시간** `(**프로젝트 직접 연관**)`
  - 제한값과 경고 타이밍이 왜 운전자 반응 시간을 고려해야 하는지 설명한다.

- **5.3.5 시스템 경계에 대한 대응** `(**프로젝트 직접 연관**)`
  - 경계 초과 시 즉시 stand-by로 가야 하며, HMI 통지가 필요하다.

- **5.3.5.1.1 급격한 활성/대기 변동 방지** `(**프로젝트 직접 연관**)`
  - 히스테리시스/채터링 방지 개념의 규정 근거로 읽으면 좋다.

- **5.3.5.5 경계 접근 시 선행 통지(lead time)** `(**프로젝트 직접 연관**)`
  - 경고를 언제 앞당겨야 하는지에 대한 근거가 된다.

- **5.3.6 제어 가능성(Controllability)** `(**프로젝트 직접 연관**)`
  - 조향 제한, 감속, 차로 위치 조정 같은 정책이 운전자에게 제어 가능한 수준이어야 한다.

- **5.3.6.3 감속 및 가속** `(**프로젝트 직접 연관**)`
  - 스로틀 제한과 감속 설계의 직접 근거다.

- **5.3.7.1 주행 차로 내 위치 유지 / 곡률에 따른 속도 적응** `(**프로젝트 직접 연관**)`
  - 속도 band, 곡률, 조향 부하를 문서에 넣을 때 참고할 수 있다.

- **5.3.7.2 기동(Manoeuvre)** `(**프로젝트 직접 연관**)`
  - 차선 변경이나 회전 같은 기동 시 운전자 이탈 경고가 있으면 안 되는 이유를 설명한다.

- **5.3.7.3 운전자 부재 반응(RMF)** `(**프로젝트 직접 연관**)`
  - ABSENT 상태 이후 안전 정지, RMF 활성화의 규정 근거다.

- **5.3.7.4 속도 제한 준수 지원** `(**프로젝트 직접 연관**)`
  - 설계 속도 범위, 현재 최대 속도, 운전자 설정 최대 속도의 관계를 이해하는 데 유용하다.

- **5.3.7.5 안전 차간 거리 지원** `(**프로젝트 직접 연관**)`
  - 안전거리/차간거리 같은 HMI 표기와 운전자 안내 방식의 근거가 된다.

---

## 4) 고장/예외 상황에서 꼭 읽을 부분

- **5.4 감지된 고장에 대한 시스템 안전 응답** `(**프로젝트 직접 연관**)`
  - 센서 막힘, 오정렬, 전기적/비전기적 고장 시 어떻게 종료해야 하는지 정의한다.

- **5.4.2.1 / 5.4.2.2** `(**프로젝트 직접 연관**)`
  - 전체 고장 시 off mode 전환과 광학적 고장 경고 요구가 있다.

- **5.4.4 일부 기능 고장 시 계속 작동 가능 조건** `(**프로젝트 직접 연관**)`
  - 일부 기능만 비활성화되는 상황을 어떻게 다룰지 판단할 때 중요하다.

---

## 5) HMI를 만들 때 특히 읽어야 할 부분

- **5.5.1 작동 모드 다이어그램** `(**프로젝트 직접 연관**)`
  - Off / On / Stand-by / Active의 관계를 먼저 잡아야 한다.

- **5.5.2 일반 요구 사항** `(**프로젝트 직접 연관**)`
  - 모드 혼동 방지의 기본 규칙이 들어 있다.

- **5.5.4.1 HMI 요구 사항 전체** `(**프로젝트 직접 연관**)`
  - 경고가 어떻게 보여야 하는지, 운전자에게 어떤 정보가 제공되어야 하는지의 핵심이다.

## 7) 한 줄 요약

- Step B는 **운전자 참여 여부와 지속시간**을 중심으로 읽고,
- Step C는 **시스템 경계/제어 가능성/감속 제한**을 중심으로 읽고,
- HMI는 **경고 시점과 안내 문구**를 중심으로 읽으면 된다.
